# iOS パフォーマンス調査 — CROW'S DESTINY

**実施済み（3案すべて実装）**: タイムステップ(dt)・GC削減・描画カリング＋エフェクト軽減

## 1. 現状調査サマリ

| 項目 | 状態 | 影響 |
|------|------|------|
| ゲームループ | rAF のみ。**dt 未使用** | FPS 低下時にゲームがスローモーション＋1フレーム遅延で飛びすぎの可能性 |
| 更新ロジック | フレーム基準（`frame++`, `x += vx`） | 端末が遅いと「遅くならない」工夫が効いていない |
| 当たり判定 | `distSquared` 使用済み ✅ | 問題なし |
| 弾 | オブジェクトプール使用済み ✅ | 敵弾は GC 負荷小 |
| パーティクル | **毎回 `new Particle`**、プールなし | burst のたびに GC 負荷 |
| 配列整理 | **毎フレーム複数 `.filter()`** | 新配列生成が多く、iOS の GC でカクつき要因 |
| 描画 | 画面外オブジェクトも**すべて描画** | カリングなしで描画負荷大 |
| エフェクト | `shadowBlur`・`globalCompositeOperation` 多用 | Canvas 負荷大（特に敵・群れカラス・雪） |
| ホーミング等 | `Math.hypot` / `atan2` / `cos` / `sin` 多用 | 負荷は中程度（二乗距離化で一部削減可能） |

---

## 2. 問題の特定

### 2-1. タイムステップ設計（iOS で効きやすい）

- **現状**: `loop(timestamp)` で `timestamp` は FPS 計測（`_adaptiveQuality`）にしか使っていない。  
  移動・タイマーはすべて「1フレーム = 1ステップ」の固定更新。
- **結果**:
  - iOS で 30fps に落ちると、見た目はスローモーションになる（1秒あたりの更新回数が半分）。
  - 逆に 1 フレームだけ 200ms かかった場合、`x += vx` が 1 回しか実行されないので「一瞬止まったように見える」だけで済むが、`dt` で積分している他ゲームのような「一瞬で飛びすぎ」は少ない。  
    ただし **「遅くならない」ための時間ベース化ができておらず、端末が遅いとゲーム全体が遅くなる**。

### 2-2. GC ・メモリ（iOS Safari でカクつきやすい）

- **毎フレームの新配列**  
  `game.js` の PLAYING / BOSS_FIGHT で、1フレームごとに例えば:
  - `this.enemies = this.enemies.filter(...)`
  - `this.crow.feathers = this.crow.feathers.filter(...)`
  - `this.flockCrows = ...`, `this.snowParticles = ...`, `this.relics = ...`, `this.obstacles = ...`
- さらに **crow.js** で `this.dashTrail = this.dashTrail.filter(...)`、**fx.js** で `floorCracks.filter`, `arenaDebris.filter`、**boss.js** で `ironWingTrail.filter`, `energyTrail.filter`, `voidAfterimages.filter`, `afterimages.filter` など。
- **パーティクル**: `FX.burst()` で毎回 `new Particle(...)` を `this.p.push()`。オブジェクトプールなし。
- **結果**: 短時間に多くの小オブジェクトが生成・破棄され、iOS の GC が走るとフレームが飛び、カクつきの主因になり得る。

### 2-3. 描画負荷

- **カリングなし**: 敵・弾・群れカラス・雪・障害物・聖遺物を「画面外」でもすべて描画。
- **重い描画**:
  - 敵スプライト: `globalCompositeOperation = 'screen'`、`shadowBlur = 6`、敵種によっては 2 回 `drawImage`（enemy3 のゴーストなど）。
  - 群れカラス: `shadowBlur = 10`、パス描画。
  - 雪パーティクル: 各粒に `shadowBlur = 5`、半透明。
- **背景**: ステージ別の `drawStageSpecific` で多数の `fillRect` / `arc` / `stroke` / `sin` 等。  
  すでに `qualityParticle` / `qualityEffect` でバースト数は減らしているが、**描画側ではパーティクル数・エフェクト強度のスキップが効いていない**。

### 2-4. その他

- **紫スキル（ホーミング羽）**: `_updateFeathersAndSkills` 内で敵・ボスとの最近傍に `Math.hypot` を多用。比較だけなら二乗距離で十分。
- **ホーミング弾**: `game.js` のボス戦で `Math.hypot`, `atan2`, `cos`, `sin` を弾ごとに実行。必要最小限なら二乗距離＋1回の `atan2` にまとめる余地あり。

## 3. 最良の修正案 3 つ

以下は、ご提示の「① iOS だと FPS が落ちる問題」と「遅くならない工夫」「描画負荷」「更新負荷」「iOS 固有」の項目を踏まえた優先度の高い 3 案です。

---

### 修正案 1: タイムステップ（dt）ベースの更新 ＋ スパイク吸収（最優先）

**目的**: rAF の実経過時間に合わせて更新し、「端末が遅くてもゲーム速度を一定に近づける」「1 フレームの遅れでキャラが飛びすぎないようにする」。

**やること**:

1. **dt の計算と上限**
   - `loop(timestamp)` 内で  
     `const dtRaw = (timestamp - this._lastLoopTime) / 1000;`  
     `const dt = Math.min(dtRaw, 0.05);` などで上限をかける（例: 50ms まで。それ以上はスパイクとして扱う）。
   - 更新メソッドには `dt` を渡す（例: `this.update(dt)`）。

2. **移動・タイマーを時間ベースに**
   - 例: `x += vx * dt * 60`（60fps を基準に正規化）、`stateT` やクールダウン系は `dt * 60` で加算。
   - 既存の「1フレーム = 1」の数値（速度・ダッシュ時間など）は、`* 60` で「1秒あたり」に換算してから `* dt` する形にすると、60fps のときは従来どおりになる。

3. **アニメ・演出**
   - アニメーションの進み方も `dt` ベースにすると、FPS が落ちても見た目が自然になる。

**参照との対応**:
- 「1フレームの時間 dt を使って、速度・移動・アニメを時間ベースにする」
- 「dt が大きすぎる時は上限をかける（スパイク吸収）」
- 「setInterval 固定更新はズレやすい → rAF + dt に寄せる」

**期待効果**: iOS で FPS が 30 前後に落ちても、ゲームの「体感速度」が維持され、ヌルヌルではなくても「遅くならない」体験に近づく。

---

### 修正案 2: GC 負荷の削減（配列の再利用 ＋ パーティクルプール）

**目的**: 毎フレームのオブジェクト生成・破棄を減らし、iOS Safari の GC によるカクつきを抑える。

**やること**:

1. **filter の削減**
   - `enemies`, `feathers`, `flockCrows`, `snowParticles`, `relics`, `obstacles` などで「active でない要素を削除」する際、**新配列を作らない**方法に変更。
   - 例: 後ろから走査し、`!active` の要素を配列末尾の要素と swap して `length--` を繰り返す in-place 削除。  
     または「削除用インデックスリスト」を 1 フレームでまとめてから一括で詰めるなど、**新しい配列を返さない**形にする。

2. **パーティクルのオブジェクトプール**
   - `FX` の `this.p` 用に、既存の `ObjectPool` と同様の仕組みを用意。
   - `burst()` では `new Particle` ではなく `particlePool.get()` で取得し、パラメータをセット。
   - パーティクルが寿命で消えるときは `particlePool.release(p)` でプールに返す（`this.p` からの削除は in-place または splice のままでも、オブジェクト自体は再利用）。

3. **その他の filter**
   - `dashTrail`, `floorCracks`, `arenaDebris`, ボスの各種 trail / afterimages なども、可能な範囲で「新配列を返さない」削除に変える。

**参照との対応**:
- 「オブジェクト生成・破棄を減らす（オブジェクトプール）」
- 「iOS Safari はメモリや GC でカクつきやすい → 確保を減らす／再利用する」

**期待効果**: フレーム間の GC が減り、特に激戦時のカクつきが和らぐ。

---

### 修正案 3: 描画負荷の削減（カリング ＋ エフェクト軽減）

**目的**: 描画コストを下げ、iOS で FPS が落ちにくくする。

**やること**:

1. **画面外カリング**
   - 敵・弾・群れカラス・雪パーティクル・障害物・聖遺物について、  
     `x < -margin || x > CFG.W + margin || y < -margin || y > CFG.H + margin` のときは **draw を呼ばない**（または draw 内で early return）。
   - マージンは 50〜100px 程度でよい（画面端で消える直前まで描画するかどうかで調整）。

2. **品質に応じたエフェクト軽減**
   - 既存の `qualityParticle` / `qualityEffect` を、**描画側**でも参照する。
   - `qualityEffect < 0.5` などなら:
     - 敵の `shadowBlur` を 0 にする。
     - `globalCompositeOperation = 'screen'` のオーバーレイ描画をスキップする。
   - パーティクルは「画面外」または「カメラから遠い」ものから描画をスキップ（すでに burst 数で減らしているので、低品質時はさらに描画数を制限する）。

3. **背景の軽量化（オプション）**
   - ステージ別の `drawStageSpecific` で、ループ回数（例: 30 → 20）や、フレームごとに描画する要素を間引く（例: 2 フレームに 1 回だけ一部を描く）など、負荷が大きいステージだけ調整する。

**参照との対応**:
- 「画面外のオブジェクトは描画しない（カリング）」
- 「影・半透明・ブラー等の重いエフェクトを減らす」

**期待効果**: 特に敵・弾・パーティクルが多い場面で FPS が持ちやすくなる。

---

## 4. 実装の優先順位（推奨）

1. **修正案 1（dt ベース ＋ スパイク吸収）**  
   iOS で「遅くならない」「変な飛び方をしない」ために最も効く。ゲームループと各 update の変更が必要だが、一度入れると他プラットフォームでも安定する。

2. **修正案 2（GC 削減）**  
   カクつきの主因になり得るため、案 1 の次に実施すると効果が分かりやすい。  
   filter の in-place 化とパーティクルプールの導入から着手。

3. **修正案 3（描画負荷削減）**  
   カリングは実装が比較的簡単で即効性がある。品質に応じた shadow/composite のスキップは、既存の `qualityEffect` と組み合わせるとやりやすい。
